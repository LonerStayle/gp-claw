# GP Claw 핸드오프 문서

> 다음 세션에서 이 문서를 읽고 컨텍스트를 빠르게 파악한 후 구현을 시작하세요.

## 프로젝트 개요

**GP Claw** — 회사 내부에서 안전하게 사용할 수 있는 AI 사무 비서.

- 엑셀, 문서, 파일 관리, Gmail 등 사무 작업을 도와줌
- 위험한 작업(파일 삭제/수정, 메일 발송)은 반드시 사용자 승인을 거침
- 회사 데이터 유출 방지를 위해 자체 호스팅 LLM 사용
- 서버리스 GPU로 비용 절감

## 핵심 설계 결정 (대화에서 확정된 것들)

### 1. LLM 선택: SK Telecom A.X 4.0 (7B + 72B 하이브리드)
- **EXAONE 4.0은 비상업 라이선스(NC)** → 제외됨
- A.X 4.0은 Qwen2.5 기반 + 한국어 CPT, **상업 이용 무료**
- KMMLU 78.3 (GPT-4o 72.5 능가), 한국어 효율 33% 높음
- **7B Light**: 단순 작업 (파일 요약, 수식 알려주기)
- **72B Standard**: 복잡 작업 (보고서 작성, 메일 작성, 계약서 검토)
- 에스컬레이션: 7B가 품질 부족하면 72B로 자동 전환

### 2. GPU: RunPod A6000 (48GB VRAM, $1.22/hr)
- 72B 4-bit AWQ: ~36GB → A6000(48GB)에서 동작 가능
- 7B 4-bit AWQ: ~4GB → 여유롭게 동작
- FlashBoot으로 콜드스타트 <2초
- L4(24GB)는 72B에 부족, A100(80GB)은 과잉 → A6000이 최적

### 3. 에이전트 구조: A+C 하이브리드
- **기본**: 단일 에이전트 (빠른 응답)
- **복잡한 요청만**: 플래너가 실행 계획을 먼저 보여줌
- 예: "이 파일 요약해줘" → 바로 실행
- 예: "엑셀 집계해서 메일 보내줘" → 계획 먼저 → 사용자 확인 → 단계별 실행

### 4. 서브에이전트 아키텍처
- 메인 세션: 대화 히스토리 + 의사결정만 담당
- 도구 실행(엑셀, 파일 등): 서브에이전트로 분리 → 결과만 반환
- **컨텍스트 절약**: 서브에이전트는 히스토리 불필요
- **보안 프롬프트 필수**: 모든 서브에이전트에 보안 규칙 주입
  - 경로 제한 (워크스페이스 외부 차단)
  - 명령어 실행 금지
  - 데이터 유출 방지
  - Dangerous 도구 직접 실행 불가

### 5. 컨텍스트/KV 캐시 관리
- System Prompt + MEMORY.md를 앞에 고정 → vLLM prefix caching 활용
- 서브에이전트는 독립 요청 → 메인 KV 캐시 오염 없음
- Pre-Compaction Flush: 컨텍스트 80% 도달 시 중요 정보를 파일에 저장

### 6. 메모리 전략 (OpenClaw 참고)
- **Working Memory**: LLM 컨텍스트 윈도우 (세션 내)
- **Daily Notes**: `memory/YYYY-MM-DD.md` (하루)
- **Long-term Memory**: `MEMORY.md` (영구)
- **검색**: 하이브리드 (sqlite-vec 70% + FTS5 30%)
- **Pre-Compaction Flush**: 컨텍스트 소진 전 자동 저장

### 7. 도구 시스템: 17개 (Safe 9 + Dangerous 8)
- Safe 도구: 승인 없이 서브에이전트가 직접 실행
- Dangerous 도구: 메인 에이전트 승인 흐름 통해서만 실행
- 승인 UI: 웹에서 도구명/파라미터/미리보기 + 승인/거부/수정 버튼
- 상세 스펙은 `docs/plans/2026-02-28-gp-claw-design.md`의 "도구 시스템" 섹션 참고

### 8. 기술 스택
| 레이어 | 기술 |
|--------|------|
| Frontend | React + TypeScript + Tailwind CSS |
| Backend | Python 3.11+ + FastAPI + LangGraph |
| LLM Serving | vLLM on RunPod Serverless (A6000) |
| Model | A.X 4.0 7B + 72B (AWQ 4-bit) |
| 실시간 통신 | WebSocket (로컬) |
| 체크포인터 | SQLite (langgraph-checkpoint-sqlite) |
| 메모리 검색 | SQLite + sqlite-vec + FTS5 |
| 파일 처리 | openpyxl, pandas, python-docx, PyMuPDF, pyhwpx |
| 이메일 | Google Gmail API (OAuth2) |

## 리서치 결과 요약 (서브에이전트에서 수집)

### RunPod vLLM 배포
- Docker 이미지: `runpod/worker-v1-vllm:v2.7.0stable-cuda12.1.0`
- OpenAI-compatible API: `https://api.runpod.ai/v2/<ENDPOINT_ID>/openai/v1`
- 환경변수: `MODEL_NAME`, `QUANTIZATION=awq`, `MAX_MODEL_LEN`, `GPU_MEMORY_UTILIZATION`
- `langchain-openai`의 `ChatOpenAI`로 바로 연결 가능

### LangGraph Human-in-the-Loop
- `interrupt()` 함수로 실행 중단 → 사용자 결정 대기
- `Command(resume=...)` 으로 재개
- **체크포인터 필수** (MemorySaver 또는 PostgreSQL)
- `thread_id`로 세션별 독립 워크플로우 관리

### FastAPI + WebSocket + LangGraph
- WebSocket이 양방향 통신 → HITL에 적합
- `graph.astream()`으로 스트리밍
- `get_stream_writer()`로 커스텀 이벤트 전송
- stream_mode: `["custom", "updates", "messages"]` 조합 사용

### A.X 4.0 모델 정보
- HuggingFace: `skt/A.X-4.0-72B` (확인 필요 — 정확한 모델 ID는 배포 시 확인)
- Qwen2.5 기반 + 한국어 CPT
- vLLM 서빙 시 `--quantization awq --enable-auto-tool-choice --tool-call-parser hermes` 권장

## 주의사항

1. **EXAONE 사용 금지** — 비상업 라이선스. A.X 4.0 사용할 것
2. **서브에이전트에 보안 프롬프트 필수** — `security.py`의 `SUBAGENT_SECURITY_PROMPT` 반드시 적용
3. **Dangerous 도구는 반드시 승인 흐름** — 직접 실행 절대 불가
4. **Phase 1 플랜의 코드는 참고용** — 테스트 실행 시 import 경로나 API가 다를 수 있으니 실제 라이브러리 문서 확인 필요
5. **A.X 4.0 정확한 HuggingFace 모델 ID** — 배포 전 HuggingFace에서 확인 필요 (skt 또는 sk-telecom 조직)
