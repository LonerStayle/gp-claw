## 1. 아키텍처 개요

### 핵심 제약 — 메시지 저장 위치
현재 메시지는 **LangGraph AsyncSqliteSaver의 체크포인트 BLOB**에만 저장된다 (`server.py:170` `aget_state`로 thread별 회수). 별도 메시지 테이블은 존재하지 않는다. 이 구조는 단일 thread 회수에는 좋지만 **모든 thread에 걸친 키워드 검색이 불가능**하다 (BLOB은 msgpack 직렬화, SQL로 효율 쿼리 불가).

### 채택 아키텍처 — Dual-store mirror

```
┌──────────────────┐     write      ┌──────────────────┐
│ WS user_message  │──────────────▶ │ AsyncSqliteSaver │ (existing — 그대로 유지)
│   handler        │                │  (checkpoints)   │
└──────────────────┘                └──────────────────┘
        │                                    │
        │ also write (mirror)                │ source of truth for replay
        ▼                                    │
┌──────────────────┐                         │
│ MessageStore     │     query       ┌──────────────────┐
│  (messages table)│ ◀─────────────  │  /search/messages │ (new REST)
└──────────────────┘                 │  /search/rooms    │
                                     └──────────────────┘
                                              │
                                              ▼
                                     ┌──────────────────┐
                                     │  Frontend        │
                                     │  ・SearchBar     │
                                     │  ・ResultsList   │
                                     │  ・SidebarFilter │
                                     │  ・Highlight+Scroll
                                     └──────────────────┘
```

체크포인트는 **유일한 진실 소스**(replay/agent 복원용)로 그대로 두고, **읽기 전용 검색 인덱스**로 `messages` 테이블을 신규 추가한다. 기존 thread는 1회 backfill 스크립트로 동기화한다.

## 3. 데이터 모델/스키마 변경

### 신규 테이블: `messages`

```sql
CREATE TABLE IF NOT EXISTS messages (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    room_id     TEXT    NOT NULL,
    role        TEXT    NOT NULL,        -- 'user' | 'assistant' | 'tool' | 'system'
    content     TEXT    NOT NULL,        -- AIMessage <tool_call> 태그는 append 시점에 정제
    created_at  TEXT    NOT NULL,        -- ISO-8601 UTC
    seq         INTEGER NOT NULL,        -- thread 내 단조 증가 (정렬 안정성)
    FOREIGN KEY(room_id) REFERENCES rooms(id) ON DELETE CASCADE
);

CREATE INDEX IF NOT EXISTS idx_messages_created_at ON messages(created_at DESC);
CREATE INDEX IF NOT EXISTS idx_messages_room_id    ON messages(room_id);
CREATE INDEX IF NOT EXISTS idx_messages_role       ON messages(role);
-- 부분일치 + 대소문자 무시는 LOWER(content) LIKE '%kw%' 패턴 사용
-- 별도 expression index는 MVP 미적용 (NFR-1: 정량 SLA 없음)
```

`rooms` 테이블은 이미 존재하므로 변경 없음. ON DELETE CASCADE로 방 삭제 시 메시지 자동 정리.

### 마이그레이션
- 신규 사용자: `MessageStore.__init__`이 `CREATE TABLE IF NOT EXISTS`로 자동 생성
- 기존 사용자: 1회성 `python -m scripts.backfill_messages` — 모든 room을 순회하며 `aget_state`로 메시지 추출 후 `messages` 테이블에 적재. 이미 일부 적재돼 있으면 멱등(`(room_id, seq)` UNIQUE 또는 사전 `DELETE FROM messages WHERE room_id=?`).

## 4. 외부 인터페이스 (REST)

### `GET /search/messages`
키워드 + 필터 조합. 페이지네이션은 `limit`/`offset`.

| 쿼리 파라미터 | 타입 | 설명 |
|---|---|---|
| `q` | string (required) | 키워드 (대소문자 무시 부분일치) |
| `room_id` | string[] (repeat) | 방 ID 다중 선택 (없으면 전체) |
| `role` | string[] (repeat) | `user`/`assistant`/`tool`/`system` (없으면 전체) |
| `from` | ISO-8601 | 날짜 시작 (포함) |
| `to` | ISO-8601 | 날짜 끝 (포함) |
| `limit` | int (default 50) | |
| `offset` | int (default 0) | |

**응답**:
```json
{
  "total": 123,
  "items": [
    {
      "id": 4711,
      "room_id": "abc123",
      "room_title": "리팩토링 회의",
      "role": "user",
      "content": "...",
      "snippet": "…키워드 주변 ±50자…",
      "match_offsets": [[12, 18]],
      "created_at": "2026-05-01T10:23:11+00:00"
    }
  ]
}
```

스니펫과 `match_offsets`는 서버에서 생성 → 프론트는 그대로 렌더 (FR-3, FR-4).

### `GET /search/rooms`
방 이름 필터 (FR-8). `q` 파라미터 하나로 부분일치 LIKE.

```json
[ {"id":"...","title":"...","updated_at":"..."} ]
```

### 기존 엔드포인트
- `GET /rooms/{room_id}/messages`: 변경 없음
- WebSocket: `user_message` 핸들러 내부에서 mirror write만 추가 (외부 프로토콜 변경 없음)

### 검색 결과 → 점프 (FR-4)
프론트엔드 라우팅이 상태 기반(`activeRoomId`)이므로:
1. 결과 클릭 → `setActiveRoomId(room_id)` + `pendingScroll={messageId, query}` 상태 세팅
2. `ChatContainer`가 mount/update 시 `pendingScroll` 발견 → 메시지 로드 후 해당 ID 요소로 `scrollIntoView` + 키워드 하이라이트 적용 → state 클리어
3. `messages` API 응답에 `id` 필드를 포함하도록 `get_room_messages` 확장 필요 (mirror table의 id를 함께 반환)

## 5. 핵심 결정 + 대안 비교

### 결정 1 — Mirror 테이블 (Dual-store) 채택
| 옵션 | Pros | Cons |
|---|---|---|
| **A. Mirror 테이블 (채택)** | SQL 인덱스 활용, 명확한 스키마, 테스트 용이 | dual-write 일관성 관리 필요, backfill 1회 |
| B. 검색 시 모든 thread `aget_state` 스캔 | 스키마 변경 없음 | O(N rooms × M msgs) 매 쿼리, 직렬화 BLOB 매번 디시리얼라이즈 |
| C. 체크포인트 BLOB에 LIKE | 코드 거의 추가 없음 | msgpack/JSON 포맷 의존 = 매우 취약, 스니펫·role 추출 불가 |

**Why A**: B는 사용자 체감 명백히 느림(NFR-1 "체감상 즉시"에 위반). C는 LangGraph 내부 포맷에 의존해 깨지면 silent fail. A는 dual-write 한 줄로 끝나고 표준 SQL이라 유지보수 쉬움. backfill은 1회성 스크립트로 충분.

### 결정 2 — LIKE vs FTS5
| 옵션 | Pros | Cons |
|---|---|---|
| **A. `LOWER(content) LIKE '%q%'` (채택)** | 추가 라이브러리/마이그레이션 없음, 한국어 토크나이징 이슈 회피 | 풀스캔(인덱스 거의 무용), 대규모에서 느려짐 |
| B. SQLite FTS5 + 한국어 트라이그램 토크나이저 | 정확한 토큰 검색, AND/OR 지원 | MVP 범위 밖(고급 쿼리는 §5 Out-of-Scope), 한국어 토크나이저 별도 작업 필요 |

**Why A**: NFR-1이 정량 SLA를 정의하지 않으며 §5에서 고급 쿼리·정규식을 명시적으로 제외했다. 사용자가 단일 사용자라 메시지 수 폭증 가능성 낮음. FTS5는 후속 PRD 자연스러운 확장점.

### 결정 3 — Dual-write 시점
| 옵션 | Pros | Cons |
|---|---|---|
| **A. WS 핸들러에서 직접 (채택)** | 명시적, 어떤 메시지가 언제 mirror되는지 단일 위치에서 추적 | 핸들러 길어짐 |
| B. LangGraph checkpointer 서브클래싱 | 자동 캡처, 구조적 우아함 | LangGraph 내부 변경에 취약, AsyncSqliteSaver 라이프사이클 복잡 |

**Why A**: B는 LangGraph 버전 업이나 internal API 변경에 매번 깨질 위험. A는 핸들러 한 군데에 명시적 `store.append(...)` 호출만 추가하면 끝.

### 결정 4 — 검색 진입점 UI 배치
| 옵션 | Pros | Cons |
|---|---|---|
| **A. 상단 SearchBar + 결과를 메인 영역 패널로 (채택)** | FR-4 점프 UX와 자연스러움(클릭 → 메인 영역이 채팅뷰로 전환) | 레이아웃 분기 필요 |
| B. 모달 다이얼로그 | 구현 가벼움 | 결과 클릭 후 모달 닫고 점프하는 흐름이 어색 |
| C. 사이드바 확장 | 사이드바 한 곳에 모든 탐색 | 결과 표시 공간 협소 |

**Why A**: FR-4는 클릭 → 방 이동 + 스크롤 + 하이라이트가 핵심. 모달이면 한 단계 더 사라지고 다시 메인이 갱신되는 깜빡임 발생. A는 한 번의 상태 전환으로 전환된다.

## 6. 위험/사이드이펙트 (preliminary, risk-annotation 카테고리)

| ID | 카테고리 | 내용 | 완화 |
|---|---|---|---|
| R-1 | **side-effect** | WS 핸들러에서 agent persist는 성공했는데 `MessageStore.append` 직전에 예외 발생 시 mirror 누락 → 검색 결과에서 빠짐 | append를 try/except로 감싸 logger.warning만 남기고 본 흐름 유지. backfill 스크립트로 정기 동기화 옵션 제공. |
| R-2 | **race** | 같은 room_id로 동시에 두 WS 세션이 user_message 보낼 때 (예: 같은 사용자가 두 탭) seq 충돌 | seq를 `MAX(seq)+1` SELECT 후 INSERT 트랜잭션으로 묶기. 또는 DB-side `ROW_NUMBER()` 대신 INSERT 시점 `created_at` 기준 정렬만 사용 (seq 제거 검토). |
| R-3 | **breaking** | 기존 사용자의 SQLite DB에 messages 테이블이 없어 첫 검색 시 빈 결과 | 앱 기동 시 자동 backfill 트리거 옵션 또는 안내 다이얼로그. `__main__.py`에서 messages 테이블이 비어 있고 rooms는 있으면 backfill 권유 로그. |
| R-4 | **perf** | LIKE는 인덱스를 사용 못해 메시지 N개에 비례한 풀스캔 → 메시지 수만 개 이상에서 느려짐 | NFR-1상 정량 목표 없으므로 MVP 허용. 후속 PRD에서 FTS5 도입 시 자연스럽게 해소. |
| R-5 | **side-effect** | AIMessage `<tool_call>` 태그가 mirror에 그대로 들어가면 사용자 검색에 노이즈 | `get_room_messages` 패턴(`tool_tag_re.sub`)을 `MessageStore.append`에서 동일하게 적용. 결과 비어있으면 append 스킵. |
| R-6 | **perf** | 검색 페이지네이션 OFFSET 기반은 깊은 페이지에서 느림 | MVP 허용 (50개 단위 무한스크롤 정도면 보통 수십 페이지 미만). 후속 개선 시 keyset 페이지네이션. |
| R-7 | **breaking** | `get_room_messages` 응답에 `id` 필드 추가(FR-4 점프용) — 프론트엔드 기존 파서/타입에 깨짐 가능 | `frontend/src/types.ts` Message 타입을 동일 PR로 업데이트, `tests/test_server.py`에 응답 스키마 회귀 테스트 추가. |
